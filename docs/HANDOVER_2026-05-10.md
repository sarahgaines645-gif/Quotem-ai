## Architecture (post-today)

### 1. Auth is default-on app-wide
`server/index.js` mounts a global middleware before the router. Every route requires sign-in unless explicitly listed in `PUBLIC_PATHS` / `PUBLIC_PREFIXES`. Public allowlist:
- `/health`, `/q-auth.js`, favicons, `/manifest.webmanifest`
- `/login`, `/signup`, `/logout`, `/forgot-password`, `/reset-password`
- `/trace-widget.js`, `/looking-glass-widget.js`
- `/assets/*`, `/widgets/*`, `/public-download/*`

Adding a new route is auto-authenticated. There is no way to forget the auth check.

### 2. User data is physically per-user
New helper `plugins/user-data.js` exposes `userDataPath(email, sub)` → `${VOLUME}/users/{email-slug}/{sub}`. Email is filesystem-safe-slugified (lowercase, non-alphanumeric → `_`).

What lives where now:
- **Threads** — `${VOLUME}/users/{slug}/threads/{thread-id}.json` (+ `{thread-id}-files/` for attachments)
- **Q's voice override** — `${VOLUME}/users/{slug}/q-voice/override.wav`
- **Generated files** (docx, images, music, video, SVG, narration) — `${VOLUME}/users/{slug}/q-generated/{token}__{filename}`

Path itself encodes the user. There is no shared file for these.

### 3. Still on owner-tag-on-record (not yet per-user-dir)
- **Scheduler jobs** — single shared `q-jobs.json`, each job carries `ownerEmail`. List/get/delete filter by owner. Not leaking, but architecturally inconsistent. Worth refactoring when there's time.
- **Memory + facts** — already per-personId via `q-memory-{id}.json` / `q-facts-{id}.json`, just under a different naming convention. Not a leak.

### 4. Confirmed safe-shared
- `q-rag.js` (bundled static knowledge base, not user content)
- `cost-tracker.js` (admin-level API cost telemetry, no per-user data)

### 5. Q's tools all carry `personEmail`
`executeTool(name, args, personId, personEmail)` and `qChat(messages, { person: req.person })` plumb the user through. Every tool that touches user data (save_situation, list_threads, read_thread, add_email_to_thread, add_note_to_thread, speak_as_q, create_document, generate_image/music/video, vectorise_image) requires `personEmail` and uses it for path resolution.
